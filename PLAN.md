## 1. 项目定位

### 一句话

**Fable 是一个以故事图（Story Graph）为核心的 AI 互动叙事引擎，兼容 SillyTavern 社区资产。**

### 与现有项目的定位对比

| | SillyTavern | RisuAI | Fable |
|--|--|--|--|
| 核心隐喻 | 聊天室 | 聊天室 | 互动小说 / TRPG |
| 数据模型 | `Message[]` | `Message[]` | `Story > Chapter > Scene > Event` |
| 角色 | 静态卡片 | 静态卡片 + 基础记忆 | 静态身份 + 动态状态 + 独立认知 |
| 世界 | Lorebook 关键词注入 | Lorebook 关键词注入 | 持久化世界状态 + 角色认知过滤 |
| 分支 | Swipe（同位置换回复） | Swipe | 剧情级分支（故事线分叉） |
| Prompt | 全量历史拼接 | 全量历史拼接 + 记忆压缩 | 精确的当前态组装 |
| 渲染 | 聊天气泡 | 聊天气泡 | 按事件类型混合排版 |

### 不做什么

- 不做通用 LLM 聊天工具
- 不做图像生成工具（有 StoryForge）
- 不重新实现 ST 的所有功能（预设管理 UI、API 测试等）
- 第一版不做多人协作（列入远期路线图）

## 2. 系统架构

### 分层设计

```
┌──────────────────────────────────────────────────────────┐
│                     展示层 (Svelte 5)                      │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌────────────┐  │
│  │ Play     │ │ Story    │ │ Character│ │ Settings   │  │
│  │ (游玩)   │ │ Overview │ │ Editor   │ │ & Import   │  │
│  └──────────┘ └──────────┘ └──────────┘ └────────────┘  │
├──────────────────────────────────────────────────────────┤
│                 叙事引擎层 (engine/ 纯逻辑)                 │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────────┐ │
│  │ SceneManager │ │ PromptBuilder│ │ StateUpdater     │ │
│  │ 场景编排     │ │ Prompt 组装   │ │ 状态更新         │ │
│  └──────────────┘ └──────────────┘ └──────────────────┘ │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────────┐ │
│  │ CharAgent    │ │ PlotGraph    │ │ ResponseParser   │ │
│  │ 角色行为     │ │ 分支图管理    │ │ LLM 响应解析     │ │
│  └──────────────┘ └──────────────┘ └──────────────────┘ │
├──────────────────────────────────────────────────────────┤
│                   兼容层 (compat/)                         │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌────────────┐ │
│  │ 角色卡   │ │ 世界书   │ │ 预设     │ │ 宏翻译器   │ │
│  │ 导入导出 │ │ 导入     │ │ 导入     │ │            │ │
│  └──────────┘ └──────────┘ └──────────┘ └────────────┘ │
├──────────────────────────────────────────────────────────┤
│                   适配器层 (adapters/)                      │
│  ┌────────────────┐ ┌────────────────┐                   │
│  │ LLM Adapter    │ │ Image Adapter  │                   │
│  │ OpenAI/Claude  │ │ ComfyUI (后期)  │                   │
│  │ Gemini/DeepSeek│ │                │                   │
│  └────────────────┘ └────────────────┘                   │
├──────────────────────────────────────────────────────────┤
│                   存储层 (db/)                              │
│  ┌──────────────────────────────────────────────────────┐│
│  │ Dexie.js (IndexedDB) + Tauri FS                      ││
│  └──────────────────────────────────────────────────────┘│
└──────────────────────────────────────────────────────────┘
```

### 关键设计原则

1. **engine/ 层为纯逻辑**：不依赖 DOM、Svelte、Tauri，可独立测试
2. **compat/ 层隔离 ST 格式**：其他模块不直接处理 ST 数据格式
3. **适配器层可插拔**：LLM/Image 适配器实现标准接口，不耦合引擎

## 3. SillyTavern 兼容策略

**原则：兼容数据，不兼容代码。导入时转换为 Fable 原生格式，运行时不保留 ST 依赖。**

| ST 资产类型 | 格式 | 兼容方式 | 优先级 |
|---|---|---|---|
| 角色卡 (V2/V3) | PNG tEXt + JSON | 完整导入，映射到 Character | P0 |
| 世界书 / Lorebook | JSON | 导入为 Location + LoreEntry + 初始 WorldState | P0 |
| 采样预设 | JSON | 导入 LLM 参数配置 + 叙事指令 | P1 |
| 上下文/指令模板 | 文本 + 宏 | 宏翻译后导入为 NarratorStyle | P1 |
| 正则脚本 | JSON | 直接导入（通用正则） | P2 |
| 插件/扩展 | JS + ST API | **不兼容**，自建扩展系统 | - |

世界书条目按语义分类转换为四种目标：
- **地点** → `Location`（含地点名词的条目）
- **角色补充** → 追加到对应 `Character.identity`
- **规则** → 注入 `NarratorStyle.customInstructions`
- **通用知识** → `LoreEntry`（运行时关键词匹配 + 角色认知过滤注入）

详细格式规范和字段映射代码见 `DESIGN.md` 第 2-3 节。

测试数据位于 `~/Opensource/projects/ai/storyforge/SillyTavern-Backup/`。

## 4. 叙事引擎核心流程

### 4.1 场景推进循环

```
┌─────────────────────────────┐
│       玩家输入/选择          │
│  (自由文本 / 点选选项)       │
└─────────┬───────────────────┘
          │
          ▼
┌─────────────────────────────┐
│     SceneManager            │
│  1. 记录玩家行为为 Event     │
│  2. 检查是否触发场景转换     │
│  3. 更新世界状态             │
└─────────┬───────────────────┘
          │
          ▼
┌─────────────────────────────┐
│     PromptBuilder           │
│  1. 组装当前态 prompt        │
│  2. 角色认知过滤             │
│  3. Lore 关键词匹配 + 注入   │
└─────────┬───────────────────┘
          │
          ▼
┌─────────────────────────────┐
│      LLM Adapter            │
│  (OpenAI / Claude / Gemini)  │
└─────────┬───────────────────┘
          │
          ▼
┌─────────────────────────────┐
│     ResponseParser          │
│  1. 解析 JSON 事件序列       │
│  2. 提取状态变更             │
│  3. 检测是否需要玩家选择     │
└─────────┬───────────────────┘
          │
          ▼
┌─────────────────────────────┐
│     StateUpdater            │
│  1. 应用 StateChange         │
│  2. 更新角色状态/关系/知识   │
│  3. 更新世界状态             │
│  4. 更新 PlotGraph 节点      │
└─────────┬───────────────────┘
          │
          ▼
┌─────────────────────────────┐
│     SceneView (渲染器)       │
│  按事件类型混合排版显示       │
│  等待下一次玩家输入           │
└─────────────────────────────┘
```

### 4.2 多角色自主行为

当玩家选择"休息"或"等待"时，引擎为每个有 active goal 的非玩家角色独立生成行为：

1. 为该角色单独构建 prompt，**只包含她知道的信息**
2. LLM 生成角色的自主行动
3. 行动产生的 StateChange 写入世界状态
4. 玩家只看到自己在场或事后得知的事件

同时生成世界事件（天气变化、战争推进、NPC 移动等）。

### 4.3 Prompt 构建 vs ST

| | ST | Fable |
|---|---|---|
| 角色信息 | 角色卡全文，不变 | 当前态快照（情绪、目标、关系值） |
| 世界信息 | 关键词触发注入原文 | 关键词触发 + 状态叠加 + 认知过滤 |
| 历史 | 尽量多塞聊天记录 | 只给当前场景最近事件 |
| 指令 | 通用 system prompt | 场景特定的生成指令（生成什么类型事件） |

Prompt 构建实现代码见 `DESIGN.md` 第 4 节。

## 5. 前端架构

### 5.1 页面结构

```
App
├── Play（主游玩界面）
│   ├── SceneView         — 场景渲染区（混合排版）
│   ├── InputBar          — 玩家输入（自由文本 + 选项按钮）
│   ├── CharacterPanel    — 侧栏：在场角色状态
│   └── MiniMap           — 侧栏：当前位置
│
├── StoryOverview（故事全局视图）
│   ├── PlotGraph         — 剧情分支图可视化
│   ├── Timeline          — 时间线（章节/场景列表）
│   └── WorldMap          — 世界地图 / 地点列表
│
├── CharacterEditor       — 角色编辑（身份 + 初始状态）
├── WorldEditor           — 世界编辑（地点、阵营、规则）
├── Settings              — LLM 配置、叙事风格、导入导出
└── Import                — ST 资产批量导入向导
```

### 5.2 场景渲染

SceneView 按事件类型使用不同组件渲染（不是统一聊天气泡）：

| 事件类型 | 渲染方式 |
|---|---|
| `narration` | 全宽散文排版，Markdown 渲染 |
| `dialogue` | 头像 + 角色名 + 情绪标记 + 对话气泡 |
| `action` | 斜体文本，角色名前缀 |
| `choice` | 卡片式选项按钮 + 可选自由输入 |
| `scene_change` | 分隔线 + 过渡文本 |
| `state_change` | 通知条（"你获得了..."） |
| `internal` | 半透明内心独白区块 |

详细组件代码在开发阶段编写。

## 7. 技术决策记录

### 为什么选 Svelte 5 而不是 React？

- 与 RisuAI 同技术栈，可以参考其 Tauri 集成模式
- Svelte 5 Runes 的响应式模型适合频繁状态更新（角色状态、世界状态实时变化）
- 更小的 bundle size，桌面端性能更好
- StoryForge 已经用了 React，两个项目技术栈互补，增加学习面

### 为什么用 Dexie.js？

- IndexedDB 封装，浏览器和 Tauri WebView 都能用
- 支持复杂查询（故事/场景/角色的关联查询）
- 支持版本迁移
- StoryForge 已验证可行

### 为什么引擎层要求纯逻辑？

- 可独立测试（不依赖 DOM / Svelte）
- 未来可抽取为独立 npm 包
- 可被其他前端框架复用
- 便于编写集成测试（模拟完整的故事推进流程）
